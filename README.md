# Привет, я Вадим 👋

### 1С/ C# / ASP.NET Core backend developer

Разрабатываю backend-приложения на **C# и ASP.NET Core**, проектирую API, работаю с PostgreSQL, Entity Framework Core и Docker. Больше всего интересуюсь архитектурой серверных приложений, авторизацией, бизнес-логикой, интеграциями и постепенным доведением учебных проектов до уровня реального продукта.

Сейчас мой основной проект — **«Доставка-Курорт»**, сервис локальной доставки для ресторанов, магазинов и торговых точек.

## Клиент-серверное взаимодействие

### REST API

Основной способ взаимодействия в проекте **«Доставка-Курорт»**:

```text
Browser → ASP.NET Core MVC Web → REST API → PostgreSQL
```

MVC-приложение выступает в роли BFF: получает запросы браузера, хранит защищённую пользовательскую сессию и вызывает API через `HttpClient`.

Через REST реализуются:

* регистрация и авторизация;
* каталог и поиск;
* профиль пользователя;
* корзина и оформление заказа;
* кабинеты продавца, курьера и администратора;
* рейтинги, отзывы, избранное и рекомендации.

### MVC и Server-Side Rendering

В клиентской части использую ASP.NET Core MVC и Razor Views. Сервер формирует HTML, а JavaScript применяется точечно: для подсказок поиска, динамических форм и отдельных интерактивных действий.

Такой подход позволяет сохранить простой клиент, централизованную серверную валидацию и возможность постепенно развивать интерфейс без обязательного перехода на отдельное SPA.

### API Gateway и взаимодействие сервисов

В учебном проекте **BookIT.Microservices** работал со структурой из нескольких сервисов и API Gateway.

В проекте выделены отдельные сервисы для:

* бронирования;
* помещений;
* ресурсов;
* администрирования;
* поддержки;
* обновления статусов.

Этот проект помог мне изучить границы сервисов, маршрутизацию запросов, независимость компонентов и сложности распределённой авторизации.

### Асинхронное взаимодействие

Изучаю событийный подход для операций, которые не должны блокировать основной HTTP-запрос:

* отправка SMS и email;
* уведомления продавцов и курьеров;
* обработка событий заказа;
* обновление поискового индекса;
* фоновые повторы запросов к внешним сервисам.

Рассматриваю:

* `BackgroundService`;
* Outbox Pattern;
* RabbitMQ;
* Kafka — для систем, где действительно нужен большой поток событий.

### Real-time взаимодействие

Планирую использовать **SignalR** в проекте «Доставка-Курорт» для:

* мгновенного обновления статуса заказа;
* уведомления продавца о новом заказе;
* появления свободного заказа у курьера;
* отслеживания доставки;
* обновления административной панели без перезагрузки страницы.

### gRPC

Изучаю gRPC как вариант внутреннего взаимодействия между сервисами, когда важны строгий контракт, высокая производительность и компактная сериализация.

Планирую использовать его не для прямого взаимодействия с браузером, а для внутренних вызовов между сервисами в будущих распределённых версиях проектов.

### Webhooks

Планирую использовать webhooks для внешних интеграций:

* подтверждение онлайн-платежа;
* события SMS-провайдера;
* интеграция с MAX;
* другие внешние уведомления.

Для таких сценариев считаю важными идемпотентность, проверку подписи и безопасную повторную обработку событий.

## Инженерные принципы

```text
Простое решение лучше сложного,
если оно закрывает задачу и оставляет пространство для развития.
```

В работе стараюсь:

* не добавлять паттерны только ради паттернов;
* не переходить к микросервисам раньше времени;
* держать бизнес-логику вне HTTP-контроллеров;
* не доверять данным клиента;
* проектировать внешние интеграции через интерфейсы;
* сохранять обратную совместимость БД;
* автоматизировать повторяемые операции;
* документировать архитектурные решения.
